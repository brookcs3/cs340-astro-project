NULLable FK Relationship (REQUIRED)

This is the ONLY major code change required

The rubric specifically requires: "At least one UPDATE sets a FK to NULL (NULLable relationship)" and "At least one relationship where a FK can be set to NULL."

Recommended Change: RunSubmissions.playerID

Instead of deleting all run submissions when a player is deleted (CASCADE), we can keep the submissions but set playerID to NULL (SET NULL). This preserves historical data.

1 Update DDL.sql 

-- Drop the existing FK constraint
ALTER TABLE RunSubmissions
DROP FOREIGN KEY fk_runsubmissions_player;

-- Make playerID nullable
ALTER TABLE RunSubmissions
MODIFY playerID INT NULL;

-- Add new FK with ON DELETE SET NULL
ALTER TABLE RunSubmissions
ADD CONSTRAINT fk_runsubmissions_player
FOREIGN KEY (playerID) REFERENCES Players(playerID)
ON DELETE SET NULL ON UPDATE CASCADE;

2 Update run-submissions.astro (Display NULL) 

<!-- In the browse table -->
<td>{row.playerName || <span class="null-value">NULL</span>}</td>

The null-value class should style it distinctly (gray, italic)

3 Update run-submissions.astro (Add Form) 

<select name="playerID">
  <option value="">-- No Player (NULL) --</option>
  {players.map(p => (
    <option value={p.playerID}>{p.displayName}</option>
  ))}
</select>

Allow empty selection which becomes NULL: const playerID = form.get('playerID') || null;

4 Update DML.sql 

-- Document the new constraint behavior
-- When a Player is deleted, their RunSubmissions.playerID becomes NULL

-- UPDATE that sets FK to NULL
UPDATE RunSubmissions
SET playerID = NULL
WHERE runSubmissionID = :runSubmissionID;

5 Rebuild and Deploy 

npm run build
npx forever restart start.cjs

2) Delete the player
3) Verify the run submission still exists with playerID = NULL
4) Verify the UI displays "NULL" for the player name
